# AGENTS.md — Atlas BidOps Project Instructions

## Role

You are the primary coding agent for integrating BidOps 招投标自动采集与人工审核系统 into the existing Atlas repository. Work as a senior .NET modular-monolith engineer and implementation planner.

## Working agreement

- Do not stop for clarification unless blocked by security, credentials, destructive actions, paid services, production data, or legal/compliance risks.
- When uncertain, make a conservative implementation choice, document it in `docs/DECISIONS.md`, and continue.
- Keep code, identifiers, classes, tables, and APIs in English.
- Keep product-facing Chinese labels where useful in UI, seed data, permissions, or menu labels.
- Prefer simple, maintainable MVP implementation over over-engineered abstractions.
- Do not implement bypasses for login, CAPTCHA, anti-bot controls, or non-public data access.
- Do not add features that help bribery, kickbacks, collusion, evasion, hidden fund flows, or non-public influence.
- Implement crawler throttling, retry limits, audit logs, kill switches, and source-level pause controls.
- AI extraction results must go to staging tables first and require human review before business import.
- One tender package can have at most one active pursuit in MVP.

## Atlas integration rules

- Use Atlas as the base framework. Do not create a new solution.
- Do not rewrite Atlas authentication, authorization, tenant isolation, logging, messaging, background task, repository, or migration infrastructure.
- Keep BidOps as an Atlas business module, preferably `src/Atlas.Modules.BidOps`, following Atlas module conventions.
- Keep the target framework and package policy aligned with Atlas. Do not upgrade Atlas to .NET 10 or any new framework in this task.
- WebApi must expose APIs, enqueue jobs, and query status only. Crawling, document extraction, AI parsing, deduplication, change detection, and long-running work must run through Atlas.Worker / Atlas background job mechanisms.
- Do not inject `AtlasTenantDbContext`, `ITenantDbContextFactory`, EF `DbContext`, or call `DbContext.Set<T>()` in BidOps business/API code. Use Atlas repositories, QueryServices, domain services, and existing data-scope conventions.
- Do not store file binary content in MySQL. Store only metadata and use `IBidOpsFileStore` with local storage for MVP and MinIO/S3-compatible storage later.

## BidOps data ownership rules

- BidOps owns its entities, entity configurations, services, queries, controllers, background jobs, AI services, crawler adapters, and compliance services.
- MVP physical storage is Atlas Tenant DB, not a separate `bidops_db`.
- Do not create a standalone `BidOpsDbContext` or a separate migration pipeline in MVP.
- All BidOps tables must use the `bidops_` prefix.
- All tenant business entities must use Atlas's current tenant isolation interface and `TenantId` rules.
- Unique indexes and deduplication keys must be tenant-scoped.
- Migrations must be generated/executed through Atlas tenant migration infrastructure.
- If Atlas does not yet support scanning module assemblies for EF `IEntityTypeConfiguration<T>`, implement a reusable module configuration scanning extension before adding BidOps tables.

## General .NET engineering rules

These common C#/.NET rules apply unless they conflict with the Atlas/BidOps rules above. Project-specific Atlas rules always win, especially the current target framework/package policy, tenant isolation, repository usage, Worker ownership of long-running work, and migration pipeline.

### Context first

- Follow the project's own conventions first, then common C# conventions.
- Before broad changes, check target framework, C# version, `global.json`, `Directory.Build.*`, `Directory.Packages.props`, project files, package policy, nullable settings, and existing tests.
- Do not change the target framework, SDK, package management strategy, or `<LangVersion>` unless explicitly requested.
- For unfamiliar syntax or generated framework code, confirm it compiles or check the relevant docs before rewriting it.
- Do not edit generated code such as `*.g.cs`, files under generated API folders, or files marked `// <auto-generated>`.

### Code design

- Prefer simple, idiomatic, maintainable C# over new layers or speculative abstractions.
- Do not add interfaces, wrappers, or abstractions unless they are needed for external dependencies, test seams already used by the project, or meaningful decoupling.
- Reuse existing services, repositories, helpers, DTO patterns, validators, and module conventions before adding new ones.
- Use least exposure: prefer `private`, then `internal`, then `protected`, then `public`.
- Keep naming, formatting, folder layout, and public API style consistent with nearby code.
- Do not add unused methods, parameters, options, or configuration keys.
- When fixing one method, check sibling methods for the same bug or contract drift.
- Comments should explain why something is necessary, not restate what the code does.
- When writing code, add necessary Chinese comments for non-obvious business rules, parsing heuristics, safety constraints, and integration tradeoffs.
- Add comments for non-obvious public methods or complex behavior when they help future maintainers.
- Keep product-facing strings localizable where the existing UI/resource pattern supports it; keep user-facing Chinese labels where useful for BidOps.

### Error handling and safety

- Guard early. Use `ArgumentNullException.ThrowIfNull(x)` for required references and `string.IsNullOrWhiteSpace(x)` for required strings.
- Avoid blanket null-forgiving `!`; prefer explicit guards, defaults, or validated construction.
- Throw precise exceptions such as `ArgumentException` or `InvalidOperationException`; do not throw or catch base `Exception` except at explicit process boundaries where logging and rethrow/translation is required.
- Do not silently swallow errors. Log with useful context and rethrow, translate, or let the error bubble according to existing project patterns.
- Do not log secrets, tokens, cookies, API keys, raw credentials, private data, or large noisy payloads unless the existing diagnostic path redacts them.

### Async and I/O

- Async methods should end with `Async`.
- Use async end-to-end; avoid sync-over-async and blocking waits.
- Do not use fire-and-forget work. If work has a timeout, cancel the underlying operation.
- Accept and pass `CancellationToken` through I/O, database, job, and loop boundaries; use cancelable delays.
- Use linked `CancellationTokenSource` plus `CancelAfter` for timeouts when appropriate.
- Use `ConfigureAwait(false)` in library/helper code when consistent with the project; omit it in app entry points and UI code unless the codebase already does otherwise.
- Stream large payloads where practical. Avoid loading large files, HTTP responses, or JSON bodies fully into memory without a clear bound.
- Prefer `Task` by default; use `ValueTask` only when the project already uses it or measurement justifies it.
- Use `await using` for async disposable resources and make ownership of streams/readers explicit.

### Performance and production readiness

- Start with simple code and optimize measured or obvious hot paths.
- Avoid unnecessary allocations in tight loops or large-payload code.
- Use structured logging with stable event context; avoid log spam.
- Use timeouts, bounded retries, throttling, and backoff for external I/O when the workflow calls for resilience.
- Prefer configuration from appsettings/environment variables over hard-coded environment assumptions.
- Keep code cross-platform unless the surrounding project or task is explicitly Windows-only.

### DTOs and immutability

- Prefer records for immutable DTOs when the existing API style supports them.
- For EF entities and Atlas DTOs, follow existing project conventions even when they use classes with setters.

## Testing guidance

- Use the test framework already present in the solution.
- Put tests in the existing relevant test project; mirror the class or feature under test where practical.
- Name tests by behavior and follow the repository's naming style.
- Use Arrange-Act-Assert structure.
- Test one behavior per test. Use parameterized tests for multiple inputs of the same behavior.
- Avoid branching and conditionals inside tests.
- Tests should run in any order and in parallel unless the existing test fixture clearly establishes shared integration state.
- Prefer testing through public APIs. Do not change visibility just for tests, and avoid new `InternalsVisibleTo` unless the project already uses that pattern for the same layer.
- Require tests for new or changed public APIs and risky business behavior.
- Use specific assertions and edge cases instead of vague success checks.
- Avoid disk I/O in unit tests. If disk I/O is necessary, use isolated/randomized paths and keep cleanup robust.
- Avoid mocks when a simple real object or in-memory implementation is clearer. Mock external dependencies, not code whose implementation is part of the solution under test.
- If FluentAssertions/AwesomeAssertions are already used in the relevant tests, prefer them; otherwise use the framework's built-in assertions.
- Use `Throws`/`ThrowsAsync` or the framework equivalent for exception assertions.

## Definition of done

For each completed phase:

- Code builds, or failure is documented with exact reason.
- Relevant tests are added or updated.
- Run available tests and record results.
- Update `docs/IMPLEMENTATION_LOG.md`.
- Update `docs/DECISIONS.md` for assumptions or tradeoffs.
- Update README or docs when behavior changes.
- Do not proceed silently past a broken migration or tenant isolation issue; record it clearly and fix if possible.
